이번 글에서는 **탐색 공간을 비싸게 만드는** 세 가지 장치(솔트, 느림, 페퍼)들이 실제 알고리즘으로 어떻게 구현되는지 본다.

> `bcrypt`, `scrypt`, `Argon2id`, `PBKDF2` 네 개를, **같은 비밀번호 `abcd123!`** 로 직접 해싱해 결과를 비교한다.

"가입 때 해싱해 저장, 로그인 때 재해싱해 비교"는 넷 다 동일하다.

```
[ 가입 ]   비밀번호 + 랜덤 솔트 ──해시──▶  (솔트·파라미터·해시) 저장
[ 로그인 ] 입력값 + (저장된 솔트·파라미터) ──해시──▶  결과 == 저장된 해시 ?
```

차이는 "해시 한 번이 얼마나 비싼가"를 만드는 방식에 있다. 그걸 중심으로 하나씩 본다.

## bcrypt
같은 `abcd123!`를 `cost 12`로 두 번 해싱한 실제 결과다.

```
저장값1: $2b$12$tx4AIrFpZhsvapteWIygSekLtzSlyxZ632u.6lIHJwFI0327fslIS
저장값2: $2b$12$6hJf9jYtucjDEmkWTlUZquZ0VqPYbLJTp52vX8uD2BY6Z/qf.OItC
```

같은 비밀번호인데 출력이 다르다. 솔트가 매번 랜덤이기 때문이다(레인보우 테이블 방어). bcrypt의 장점은 **모든 정보를 한 문자열에 담는다**는 점이다.

```
$2b$12$tx4AIrFpZhsvapteWIygSe kLtzSlyxZ632u.6lIHJwFI0327fslIS
 │  │  └─────── 솔트(22자) ──┘ └────────── 해시(31자) ──────┘
 │  └ cost = 12 (작업강도)
 └ 알고리즘 버전 2b
```

`DB`에는 이 60자 문자열 하나만 저장하면 된다. 솔트가 그 안에 평문으로 들어 있으니, 검증할 때 따로 꺼낼 컬럼이 필요 없다.

`bcrypt`의 방어 축은 **시간 하나**다. `cost`를 `1` 올릴 때마다 해싱 시간이 2배가 된다.

| cost | 시간 |
|---|---|
| 8 | 17.5 ms |
| 10 | 69.3 ms |
| 12 | 277.4 ms |
| 14 | 1110.6 ms |

`cost 12`면 정상 로그인은 `0.3초`로 끝나지만, 공격자가 수십억 개를 돌리려면 그 `0.3초`가 곱해져 비현실적이 된다.

`bcrypt`는 입력을 **72바이트까지만** 본다. 그 이상은 무시되거나(구버전) 에러가 난다(최신 라이브러리). 그래서 긴 비밀번호를 대비해 보통 입력을 `SHA-256`으로 먼저 줄여서 넣는다. 이 선처리는 뒤에 나올 페퍼 적용과 자연스럽게 합쳐진다.

## scrypt
`bcrypt`는 메모리를 거의 안 썼다. `scrypt`의 핵심은 **memory-hard**, 한 번 해싱할 때 메모리를 통째로 점유한다.

```
저장값1: $scrypt$ln=14,r=8,p=1$y7LHtQHaO5ic07CViwHJAA==$/oyfIaMj6R5P17qF/QDCSqE6Cf/mnwxqauLVfmDMn/Y=
저장값2: $scrypt$ln=14,r=8,p=1$V7CyfkV8KRZgX1Sy8VO/HQ==$DkMvVK+FJlQMe3JQeBZAxTTI77zaf3I33CYv3ZOWe+c=
```

`scrypt`가 쓰는 메모리는 공식으로 정해져있다.

```
메모리 = 128 × N × r = 128 × 16384 × 8 = 16 MiB
```

저장 형식은 `bcrypt`와 다르다. **표준 단일 문자열이 없어서** 파라미터(N/r/p)를 직접 박아 저장해야 한다.

```
$scrypt$ln=14,r=8,p=1$y7LHtQHaO5ic07CViwHJAA==$/oyfIaMj6R5P17qF/QDCSqE6Cf/mnwxqauLVfmDMn/Y=
 │       │            │ 솔트(24자)              └ 해시(44자)
 │       └ ln=log2(N)=14 (N=16384), r=8, p=1
 └ 알고리즘
```

`bcrypt`의 "시간"에 **"메모리"가 추가**된 형태다. `N`을 키우면 시간과 메모리가 **같이** 늘어난다.

| N | log2 | 메모리 | 시간 |
|---|---|---|---|
| 4096 | 12 | 4 MiB | 11.8 ms |
| 8192 | 13 | 8 MiB | 23.5 ms |
| 16384 | 14 | 16 MiB | 47.9 ms |
| 32768 | 15 | 32 MiB | 184.1 ms |
| 65536 | 16 | 64 MiB | 375.5 ms |

`GPU/ASIC`는 코어가 수천 개지만 **코어당 메모리는 아주 적다.** `bcrypt`(메모리 거의 0)는 그 수천 코어로 동시에 수천 개를 추측할 수 있지만, `scrypt`는 추측 하나당 `16 MiB`가 필요해 동시 추측 수가 확 줄어든다. 공격자의 가장 큰 무기인 **대규모 병렬화를 메모리로 틀어막는** 것이다.

대신 운영 부담이 생긴다. 서버가 동시 로그인마다 그만큼 메모리를 쓰므로, 메모리 용량과 동시 접속량을 보고 튜닝해야 한다. 비밀번호 저장이면 `OWASP`는 `N`을 더 높게(예: 2¹⁷ ≈ 128 MiB) 권장한다.

## Argon2id
`Argon2`는 `scrypt`의 `memory-hard`를 계승하면서 두 가지를 더 손봤다. **변종 선택**과 **파라미터 독립 조절**이다.

```
저장값1: $argon2id$v=19$m=65536,t=3,p=4$lguRxe837zTduw1ozdHZGA$09/Ly3paxYHvBHh8/BGiq55Ot4PDRDgeCxNrAyL6S9w
저장값2: $argon2id$v=19$m=65536,t=3,p=4$KMTjFfhfakcEzTdNEbGBXw$dhCA6SegzpEZ9EVqzkxQ7fcx5+6RvjjMJrCHst/Vs1s
```

`scrypt`와 달리 **bcrypt처럼 깔끔한 표준 PHC 문자열**을 라이브러리가 알아서 만든다.

```
$argon2id$v=19$m=65536,t=3,p=4$lguRxe837zTduw1ozdHZGA$09/Ly3paxY...
 │        │    │              │ 솔트          └ 해시
 │        │    └ m=메모리(KiB), t=반복(시간), p=병렬도
 │        └ 버전 v=19 (Argon2 1.3)
 └ argon2id (i와 d를 섞은 하이브리드)
```

### d / i / id
`scrypt`엔 없던 개념이다. 메모리 접근 방식에 따라 셋으로 나뉜다.

- Argon2d: 메모리 접근이 데이터에 의존 → `GPU` 저항은 강하지만 사이드채널(타이밍) 공격에 취약. 암호화폐 `PoW`용.
- Argon2i: 메모리 접근이 데이터와 무관 → 사이드채널엔 안전하지만 `GPU` 저항이 약간 약함.
- Argon2id: 첫 패스는 `i`, 나머지는 `d`로 도는 **하이브리드**. 둘의 장점을 합친다.

그래서 비밀번호 저장이면 **무조건 Argon2id**다. `RFC 9106/OWASP` 모두 이걸 기본 권장으로 못 박았다.

### m / t / p 독립 조절
scrypt는 `N` 하나로 시간과 메모리가 **묶여서** 움직였다. `Argon2`는 셋을 **따로** 돌린다.

메모리(m)만 올릴 때 (t=3, p=4 고정):

| 메모리 | 시간 |
|---|---|
| 16 MiB | 38.5 ms |
| 32 MiB | 98.0 ms |
| 64 MiB | 165.6 ms |
| 128 MiB | 712.4 ms |

반복(t)만 올릴 때 (메모리 64 MiB 고정):

| 반복 t | 시간 |
|---|---|
| 1 | 77.1 ms |
| 2 | 125.4 ms |
| 4 | 206.6 ms |
| 8 | 393.3 ms |

"메모리는 서버 여유만큼만, 강도는 시간으로 더" 같은 정밀 튜닝이 가능하다. `scrypt`보다 운영이 유연해진 핵심 지점이다.

>  **m을 키운다고 출력 해시가 커지지 않는다.** m은 계산 도중 굴리는 작업 메모리(비용)이고, 출력 크기는 `hash_len`이라는 별개 파라미터(보통 32바이트)가 정한다. m을 16→128 MiB로 8배 키워도 출력은 계속 32바이트다.

또한 라이브러리에 `check_needs_rehash()`가 내장돼 있다. 옛 파라메터(m=8192, t=1, p=1)로 만든 해시를 더 강한 정책에 넣으면 "재해싱 필요(True)"가 리턴된다. **로그인 성공 순간 이 신호로 새 파라메터로 교체**하면, 파라미터 업그레이드가 코드 한 줄로 된다.

## PBKDF2
앞의 셋과 결이 다르다. `PBKDF2`는 `memory-hard`가 아니라, 사실상 **HMAC을 수십만 번 반복하는 루프**다.

```
저장값1: pbkdf2_sha256$600000$PXcupiK4VfbRrNfSMXhIAQ==$ngmh4FD9Lkpu30yOaCgxr3ivpQbqcCNlTp5CPK//6U0=
저장값2: pbkdf2_sha256$600000$bF5IfD4q9TLAOfMmK9QzDQ==$v8nIPfe6m9OcdiRWPNzLY3ApbaxlZO86MH3Z4MpPTig=
```

저장 형식은 `Django`의 관례가 사실상 표준처럼 쓰인다.

```
pbkdf2_sha256$600000$PXcupiK4VfbRrNfSMXhIAQ==$ngmh4FD9Lkpu30yOaCgxr3ivpQbqcCNlTp5CPK//6U0=
 │            │       │ 솔트(24자)             └ 해시(44자)
 │            └ 반복횟수 600,000 (work factor)
 └ 알고리즘_PRF(sha256)
```

방어 축은 **반복횟수(시간) 하나뿐**이다. 메모리는 거의 안 쓴다.

| 반복 | 시간 |
|---|---|
| 100,000 | 58.7 ms |
| 300,000 | 176.4 ms |
| 600,000 | 346.8 ms |
| 1,200,000 | 708.5 ms |

축이 시간 하나뿐이라는 건, **GPU/ASIC 병렬 공격을 막을 메모리 장벽이 없다**는 뜻이다. 그래서 네 알고리즘 중 현대 크래킹 하드웨어에 **가장 약한 게 PBKDF2**다. 반복수를 아무리 올려도 `memory-hard`만큼의 병렬 방어는 안 된다.

그런데도 살아남은 이유는 **표준 승인**이다. `PBKDF2`는 `NIST SP 800-132 / FIPS 140` 승인 알고리즘이다. 반면 `bcrypt/scrypt/Argon2`는 (보안성과 별개로) 이런 정부 표준 목록에 없다.

> 규제 환경에서는 "더 안전한 알고리즘"이 아니라 "**승인된** 알고리즘"을 써야 하는 상황이 생긴다. 특히 국내 공공기관은 KCMVP(암호모듈 검증)나 승인 암호알고리즘 사용을 요구받는 경우가 많은데, 그 목록엔 `SHA-256`/HMAC/PBKDF2는 들어가지만 Argon2/bcrypt는 보통 빠져 있다. 게다가 PBKDF2는 JDK 표준(`PBKDF2WithHmacSHA256`)에 내장돼 폐쇄망에 외부 라이브러리를 추가할 필요도 없다. Argon2id가 기술적으로 더 좋다는걸 알면서도 PBKDF2가 현실적 답이 되는 환경이 분명히 있다.

조금이라도 더 단단하게 하려면 `PRF`를 `SHA-512`로 쓰는 선택지가 있다. `GPU`가 64비트 연산에 비효율적이라 약간의 저항이 더 붙는다. `OWASP` 권장 반복수는 `SHA-256` 기준 60만 회다.

## 공통 처리 — 페퍼와 선처리
네 알고리즘 모두에 **동일하게** 적용되는 지점이 있다. 1편에서 본 **페퍼**다. 보통 `HMAC-SHA256(페퍼, 비밀번호)`을 먼저 돌리고 그 결과를 해시 함수에 넣는다. 이 한 단계가 두 가지를 동시에 해결한다.

1. 페퍼 적용 — 페퍼는 `DB`가 아니라 `HSM/KMS`에 둔다. `DB`가 통째로 털려도 페퍼가 안 털리면 공격자는 무차별 대입을 시작조차 못 한다.
2. 길이 고정 — `HMAC` 출력은 항상 고정 길이라, `bcrypt`의 `72바이트` 제한도 자연스럽게 회피된다.

즉 "어떤 알고리즘을 쓰든" 위에 페퍼를 얹는 구조는 같다. 알고리즘 선택과 페퍼는 독립적인 결정이다.

## 환경별 선택 기준
네 개를 한자리에 놓으면 이렇게 갈린다.

| | bcrypt   | scrypt | Argon2id | PBKDF2 |
|---|----------|---|---|---|
| 등장 | 1999     | 2009 | 2015 (PHC 우승) | 2000 |
| 방어 축 | 시간       | 시간 + 메모리 | 시간/메모리/병렬(독립) | 시간 |
| memory-hard | X        | O | O | X |
| GPU/ASIC 저항 | 중        | 상 | 상 | 하 |
| FIPS 승인 | X        | X | X | O |
| 저장 문자열 | 자체 표준    | 표준 없음(직접) | 표준 PHC | 관례(Django) |
| 특이사항 | 72바이트 제한 | 메모리 운영 부담 | 변종, 자동 재해싱 | JDK 내장 |

선택은 보안 강도순이 아니라 **제약 조건**을 따라간다.

```
신규 시스템이고 환경(의존성/인프라)을 통제할 수 있나?
├─ 예 ───────────────▶ Argon2id   (현시점 1순위)
└─ 아니오
   ├─ FIPS/규제/폐쇄망 제약이 있나? (공공/금융)
   │   └─ 예 ─────────▶ PBKDF2     (승인 알고리즘, JDK 내장)
   ├─ memory-hard가 필요한데 Argon2를 못 쓰나?
   │   └─ 예 ─────────▶ scrypt
   └─ 광범위 지원/검증된 무난한 기본값이면 충분 ▶ bcrypt
```

핵심은 "가장 안전한 게 항상 정답은 아니다"라는 점이다. `Argon2id`가 기술적으로 가장 앞서지만, **규제/폐쇄망 환경에서는 승인 여부와 운영 제약이 먼저**다.

내가 일하는 공기업 환경(폐쇄망, KCMVP 요건, JDK 의존성 최소화)에서는 `GPU` 저항이 가장 약한 `PBKDF2`가 오히려 현실적 정답이 된다. 거기에 `PRF`를 `SHA-512`로 올리고, 솔트(기본)와 페퍼(별도 보관)를 얹는 것이 "내 환경에서 고를 수 있는 최선"이다.

## 정리
- 네 알고리즘은 "해시 한 번을 비싸게 만드는 방식"이 다르다. **bcrypt/PBKDF2는 시간**, **scrypt/Argon2id는 시간 + 메모리(memory-hard)**.
- `memory-hard`가 `GPU/ASIC` 병렬 공격을 막는 핵심이고, 그래서 **PBKDF2가 가장 약하다.**
- 그럼에도 **FIPS 승인 + JDK 내장** 덕에 `PBKDF2`는 규제/폐쇄망에서 살아남는다. 보안 강도와 선택 기준은 별개다.
- 신규 시스템의 일반 정답은 **Argon2id**, 검증된 무난한 기본값은 **bcrypt**, 메모리 방어가 필요하면 **scrypt**, 규제/폐쇄망이면 **PBKDF2**.
- 어떤 걸 고르든 **솔트 + 느림 + 페퍼**라는 세 장치는 그대로 들어간다. 알고리즘 선택은 그 위의 한 층일 뿐이다.

티빙 `CI/DI` 유출에서 시작한 질문이, 결국 "내 손으로 비밀번호를 어떻게 저장할 것인가"라는 결정으로 이어졌다. 개념(1편)과 구현/선택(2편)을 둘 다 거치고 나니, 막연히 "제일 좋은거 쓰면 되겠지" 하던 것이 "왜, 무엇을, 어떤 제약 아래에서"로 바뀌었다.
