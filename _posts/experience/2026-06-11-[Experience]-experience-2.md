## 들어가며
어떤 사이트의 예약 페이지를 들여다볼 일이 있었다. 화면에 보이는 예약 가능 시간표가 어디서 오는지 궁금해서 개발자도구 Network 탭을 열었는데, 페이지 주소로 직접 호출해도 정작 데이터는 안 왔다. HTML 껍데기만 오고, 실제 시간표 데이터는 페이지가 로딩되면서 **따로 호출하는 다른 요청**에 담겨 있었다.

그 요청을 찾아보니 처음 보는 형태였다. 엔드포인트는 `/graphql` 하나인데, 요청 본문(body)에 "내가 받고 싶은 필드들"을 길게 적은 쿼리 문자열이 들어 있었다. 응답 `JSON`은 그 쿼리에 적은 필드 모양 그대로 돌아왔다.

`REST`만 써온 입장에선 처음 보는 방식이었다. 그래서 이 글은 그날 `GraphQL`이 뭔지 파고들면서 정리한 내용이다.

> 스키마나 쿼리 블록의 필드들은 모두 예시이다.

## GraphQL은 '방식'이다, 도구가 아니다
`GraphQL`은 그 자체로는 **규격(방식)**이다. `REST`가 "아키텍처 스타일"인 것과 같은 맥락이다. 질의 문법, 스키마 문법(`SDL`), 응답 형태(`data`/`errors`) 같은 게 `GraphQL` 표준에 정의돼 있다.

다만 그 규격을 실제 코드로 돌리려면 **그것을 구현한 라이브러리**가 필요하다. 자바라면 `graphql-java`, `JS`라면 `Apollo·graphql-js` 같은 것들이다.

| 구분 | 속하는 것 |
|------|-----------|
| **방식(규격)** | 질의 문법, 스키마 문법(`SDL`), 응답 형태 → 언어/구현체 무관하게 동일 |
| **도구(구현체)** | `graphql-java`, `Apollo` 등 라이브러리 → 그 규격을 특정 언어에서 굴리는 수단 |

`SQL`이 규격이고 그 문법은 어디서나 같지만, 그걸 실행하는 `JDBC` 드라이버나 `MyBatis`는 도구인 것과 정확히 같은 구조다. `SDL`은 `SQL` 문법 쪽, `graphql-java`는 `MyBatis` 쪽이다.

## REST와의 차이점

### 엔드포인트: 여러 개 vs 하나
`REST`는 자원마다 주소가 갈린다. `/products/1`, `/products/1/schedule`, `/products/1/menu` 처럼 경로로 자원을 지정한다. `GraphQL`은 **엔드포인트가 하나(graphql)** 이고, 무엇을 원하는지는 경로가 아니라 **요청 본문의 쿼리**로 표현한다.

### 응답의 모양을 누가 정하는가
이게 본질적인 차이다.

`REST`에서는 **서버가 응답의 모양을 고정**한다. 클라이언트는 서버가 주는 대로 받을 수밖에 없다. 이로 인해 두 가지 문제점이 생긴다.
- **오버페칭**: 필드 `3`개만 필요한데 서버가 `20`개를 줘서 나머지를 버린다.
- **언더페칭**: 한 화면을 그리는 데 필요한 데이터가 여러 엔드포인트에 흩어져 있어서, **요청을 여러 번 왕복해야 한다.**

`GraphQL`은 **클라이언트가 필요한 필드를 쿼리로 골라 보낸다.** 그러면 서버는 딱 그 필드만 담아 응답한다. 예약 화면 하나를 그릴 때 매장 정보, 예약 일정, 메뉴가 다 필요하다면 `REST`에선 세 번 왕복할 것을 한 쿼리에 묶어 한 번에 받는다.

```graphql
query {
  schedule(input: { productId: "...", startDate: "...", endDate: "..." }) {
    timeSlots {
      startTime
      bookedCount
      isOnSale
    }
  }
}
```

응답 `JSON`은 이 쿼리 모양 그대로(`data.schedule.timeSlots[]`에 적은 필드들만) 돌아온다. **쿼리의 모양이 곧 응답의 모양**이다.

요청 본문에는 보통 세 가지가 같이 들어간다.
- `query`: 어떤 필드를 받을지 (필드 선택)
- `variables`: 그 쿼리에 넣을 입력값 (ID, 날짜 범위 등)
- `operationName`: 쿼리 이름

## 서버는 이걸 어떻게 처리하나
서버 입장에서 `GraphQL` 처리는 크게 두 조각이다. **스키마(계약서)** 와 **리졸버(실제 처리 함수)**.

**스키마(`SDL`)** 는 "이런 타입과 필드를 제공한다"는 계약이다. 클라이언트가 쿼리에 적을 수 있는 필드의 전체 목록이 여기 정의된다. 그러나, 스키마가 곧 `DB` 컬럼 전부의 노출은 아니다. 도메인 관점에서 "클라이언트가 봐도 되는" 것만 모델링해 올린다.

```graphql
type TimeSlot {
  startTime: String
  bookedCount: Int
  totalStock: Int
  isOnSale: Boolean
  isOpen: Boolean
}
```

**리졸버**는 특정 쿼리 필드가 들어왔을 때 실제 데이터를 만들어 반환하는 함수다. 내부적으로 `DB`를 조회하든 다른 서비스를 부르든, 그건 평소 백엔드 로직과 똑같다.

여기서 `REST`와 결정적으로 다른 메커니즘이 하나 있다. 리졸버는 객체를 **통째로 채워서 반환**하는데, 응답 `JSON`에는 **클라이언트가 쿼리에 적은 필드만** 담긴다. 라이브러리(graphql 엔진)가 응답을 직렬화할 때 쿼리를 보고 필요한 필드만 골라내기 때문이다. 그래서 같은 엔드포인트라도 클라이언트가 필드 2개만 달라면 2개만, 전부 달라면 전부 온다. **서버 코드는 그대로인데 응답 모양이 요청에 따라 달라진다.**

```
POST /graphql
  → 엔진이 query(필드 목록) + variables(입력값) 파싱
  → 해당 필드의 리졸버 실행 → 데이터 통째로 반환
  → 엔진이 요청된 필드만 남겨 data로 응답
```

## GraphQL의 장점
요청 횟수를 줄여준다. 하지만 대형 서비스가 `GraphQL`을 택하는 진짜 이유는 **다양한 클라이언트 대응**이다.

같은 매장 데이터를 모바일 앱은 상세하게, PC 웹은 더 풍부하게, 지도 안 미니뷰는 이름과 썸네일만 필요로 한다. 같은 데이터인데 클라이언트마다 원하는 필드가 다르다.

`REST`는 여기서 두 가지 선택지가 존재한다.
1. 모든 걸 다 주는 엔드포인트 하나를 두면 화면마다 안 쓰는 데이터를 모두 받아 버린다(오버페칭).
2. 화면마다 전용 엔드포인트를 만든다. 엔드포인트가 엄청나게 늘어나며 화면이 바뀔(수정) 때마다 **백엔드를 수정하고 배포**해야 한다. 프론트가 필드 하나 더 필요할 때마다 백엔드 팀 일정에 묶인다.

`GraphQL`은 백엔드가 **스키마 하나**만 노출하고, 그 안에서 클라이언트가 필요한 필드를 알아서 골라 쓴다.
- **백엔드 배포 없이 프론트가 자율적으로 화면을 구성**한다. 스키마에 이미 있는 필드라면 새 화면 추가도 프론트의 쿼리 변경만으로 끝난다.
- 오버페칭, 언더페칭을 동시에 해결한다.
- 엔드포인트 관리가 쉬워진다. 스키마 하나로 여러 종류의 클라이언트를 대응한다.

**BFF(Backend For Frontend)** 패턴과 같은 효과를 낸다. `GraphQL`은 `BFF`를 **클라이언트가 셀프서비스로 하게 만든, 일반화된 BFF**라고 볼 수 있다.

## GraphQL의 단점
물론 단점 또한 존재한다. 프론트의 편의를 얻는 대신 **복잡도가 서버로 넘어온다.**

1. **캐싱이 어렵다.** `REST`는 `GET URL`마다 브라우저, `CDN`, 프록시가 자연스럽게 캐싱한다. `GraphQL`은 한 `URL`로 `POST`를 보내니 그게 안 된다. 캐싱을 따로 설계해야 한다.
2. **보안과 `DoS`에 취약해진다.** 단일 엔드포인트가 클라이언트가 짠 깊은 쿼리를 받다 보니, 악의적으로 무겁게 중첩된 쿼리로 서버에 과부하를 줄 수 있다. 쿼리 깊이 제한, 비용 분석, 허용 목록(`persisted query`) 같은 방어를 추가로 깔아야 한다.
3. **모니터링과 에러 처리가 까다롭다.** 트래픽이 한 `URL`에 몰려서 엔드포인트별 메트릭, 로깅, `rate limit`등 을 잡기 어렵다. 게다가 `GraphQL`은 에러가 나도 보통 `HTTP 200`으로 응답하고 본문 `errors`에 담는다. `HTTP` 상태코드 기반으로 돌아가던 기존 모니터링과 알람이 그대로는 안 먹힌다.
4. **`DB` 부하가 자동으로 줄지는 않는다.** `GraphQL`의 필드 선택은 응답을 자르는 시점이 **`DB` 조회 이후**다. 기본 구현은 리졸버가 객체를 통째로(사실상 `SELECT *`) 가져온 뒤 메모리에서 잘라낸다. 즉 클라이언트가 2개 필드만 달라고 해도 `DB`는 풀 로우를 다 긁어온다. **컬럼 절약 효과는 서버 - 클라이언트 네트워크 구간이지, `DB` 부하가 아니다.** 이걸 줄이려면 요청된 필드(`selection set`)를 보고 맞춤 쿼리를 만들거나, 무거운 필드를 별도 리졸버로 떼어 요청 시에만 조회하게 하는 등 추가 작업이 필요하다.

**그리고 진짜 골칫거리는 N+1이다.** 중첩 데이터(`매장 { 메뉴 { 옵션 } }`)를 요청하면 매장 1번 → 메뉴마다 옵션 조회 → 메뉴 30개면 옵션 쿼리 30번… 식으로 쿼리가 폭발한다. 이걸 막으려고 `DataLoader` 같은 배칭(한 번에 `IN`절로 묶어 조회)을 또 얹어야 한다.

**단순한 서비스에서는 이 모든 복잡도가 손해다.**

## 그래서 언제 쓰는가
판단 기준은 단순하다.

- **단순 단건/리스트 조회(예: 메뉴 목록)** → `REST`가 압도적으로 낫다. `GraphQL`의 장점이 쓸 데가 없고 복잡도만 늘어난다.
- **한 화면이 여러 자원을 조합하거나, 클라이언트 종류가 다양한 경우** → `GraphQL`이 좋다. 백엔드를 안 건드리고 화면마다 필요한 모양을 맞출 수 있다.

하지만 `GraphQL`이 "백엔드 작업을 없애 준다"는 건 아니다. 프론트의 셀프서비스는 어디까지나 **스키마에 이미 노출된 필드 안에서만** 작동한다. 스키마에 아예 없는 데이터를 새로 추가해야 하면, 스키마에 필드를 더하고 리졸버를 구현하고 배포해야 한다.

찾아보니 현재 실제 업무환경에서도 `GraphQL`은 많이 발전했지만 `REST`를 대체하지 않았다. 여전히 `REST`가 주류이고, 대형 서비스들도 복잡한 클라이언트용엔 `GraphQL`, 단순 내부 서비스엔 `REST`를 쓰는 **하이브리드**를 유지한다.

**결국 "둘 중 뭐가 낫냐"가 아니라 "어디에 뭘 쓰냐"의 차이다.**

## 마무리
`REST`만 쓰다 보니 "응답은 서버가 정하고 클라이언트는 받는다"가 너무 당연해서, 그게 하나의 선택이라는 걸 의식하지 못했다. `GraphQL`은 그 선택을 뒤집어 **응답의 모양을 클라이언트에게 넘긴** 방식이다.

그 대가로 캐싱, 보안, 모니터링, 서버 복잡도라는 비용을 떠안는다. 결국 `REST` 의 진화가 아닌, 클라이언트가 다양하고 화면마다 데이터가 제각각인 상황에 특화된 새로운 도구다. 우연히 마주친 `GraphQL`이 왜 이런 방식이 존재하고 어디에 어울리는지까지
알게 되었다.
